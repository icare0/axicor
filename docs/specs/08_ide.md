# 08 — Genesis IDE

> Интегрированная среда разработки и наблюдения для Genesis-симуляций.
> Построена на **Bevy**. Концептуально наследует оконно-воркспейсную модель **Blender**.

---

## Видение

Genesis IDE — это не текстовый редактор. Это **среда наблюдения и управления нейросетью в реальном времени**. Главная метафора: ты смотришь на живой мозг через разные «линзы» — каждая панель это отдельная линза с собственной семантикой.

Цель: чтобы исследователь мог одним взглядом охватить состояние шарда, запустить симуляцию, покопаться в конфиге, и тут же увидеть эффект в 3D — без переключения между терминалами, файлами и графиками.

---

## Уровень 1 — Высокая Абстракция

### 1.1 Общие принципы

| Принцип | Смысл |
|---|---|
| **Воркспейс-ориентированность** | Интерфейс состоит из разделяемых/объединяемых панелей — как в Blender. Нет фиксированного layout'а. |
| **Данные правят UI** | Состояние приложения живёт в ECS Bevy. UI — это observer поверх данных. |
| **Не блокирующее** | Симуляция и UI — независимые системы Bevy. Запущенная сеть не тормозит редактор. |
| **Конфиг как источник правды** | TOML-конфиги (simulation.toml, anatomy.toml, blueprints.toml) — единственный способ задать архитектуру. IDE их читает, визуализирует и редактирует. |
| **Zero-Copy не нарушается** | IDE не должна нарушать инварианты движка. Визуализация работает с read-only snapshot'ами VRAM-состояния. |

---

### 1.2 Типы Панелей (Editors)

По аналогии с Blender, каждое окно имеет **тип**. Пользователь сам собирает layout из нужных типов.

| ID | Название | Что показывает |
|---|---|---|
| `world_view` | **3D World View** | Пространственная визуализация нейронов, аксонов, шардов в 3D |
| `config_editor` | **Config Editor** | TOML-редактор с валидацией и схемой |
| `timeline` | **Timeline** | Тики симуляции, Day/Night фазы, события |
| `neuron_inspector` | **Neuron Inspector** | Детали выбранного нейрона/аксона/синапса |
| `shard_map` | **Shard Map** | 2D вид планарного шардинга, загрузка шардов |
| `signal_scope` | **Signal Scope** | Осциллограф спайков, активность популяций |
| `log_console` | **Log Console** | Системные логи, предупреждения Baker'а |
| `bake_panel` | **Bake Panel** | Запуск Baker'а, прогресс компиляции |

> На старте реализуется только `world_view`. Остальные — по мере роста движка.

### 1.4 Жизненный цикл сессии

```
Запуск IDE
    │
    ▼
[Runtime Discovery]
    │
    ├── Сканировать диапазон портов (default: 7700–7800)
    │   ├── Нет ответа → предложить запустить runtime самостоятельно
    │   ├── Один runtime найден → подключиться автоматически
    │   └── Несколько runtime найдены → показать список (IP:port, Shard ID, статус)
    │                                    пользователь выбирает цель
    ▼
[Подключено к Runtime (или «автономный» режим)]
    │
    ├── [Connected] Получить первый snapshot → визуализировать в World View
    │
    ├── [Автономный] Открыть конфиг-папку → валидировать → Baker → запустить runtime
    │
    ▼
[Live-наблюдение]
    │
    └── Стоп / отключиться / сменить runtime / редактировать / пересобрать / повторить
```

**Runtime Discovery — детали:**

| Сценарий | Поведение |
|---|---|
| Нет runtime'ов | Стартовый экран с кнопками «Открыть конфиг» и «Подключиться вручную» |
| Один runtime | Автоподключение + уведомление в StatusBar |
| Несколько runtime | Модальное окно со списком: порт, Shard ID, кол-во нейронов, фаза (Day/Night) |
| Runtime недоступен после подключения | Переход в «Disconnected» режим, попытки переподключения с бэкофом |

> Каждый genesis-runtime объявляет себя на фиксированном порту (конфигурируется в `simulation.toml` → `[ide] port = 7700`). Discovery — это простой TCP-ping по диапазону + handshake-пакет с метаданными шарда.

## Уровень 2 — Ключевые Подсистемы

### 2.1 Оконная система (Workspace Engine)

Вдохновлена Blender's Area/Region/Editor system:

- **Area** — прямоугольная зона экрана. Содержит ровно один Editor-тип.
- **Split** — пользователь делит Area на две (горизонтально или вертикально). Неограниченная глубина.
- **Merge** — обратная операция: две соседние Area объединяются в одну.
- **Header** — верхняя полоска внутри Area. Переключатель типа Editor + контекстные кнопки.

Хранение: все Areas — ресурс Bevy (`WorkspaceLayout`). Layout сериализуется в `workspace.toml` и восстанавливается при следующем запуске.

### 2.2 3D World View (MVP)

Первая и единственная реализуемая панель на старте.

---

#### Визуальные Сущности и Ассеты

**Решение: стандартные Bevy-примитивы без кастомных шейдеров (на старте).**

Причина: morphing не нужен, пишем кастомные шейдеры только когда базовые средства упрутся в потолок. Bevy умеет `Mesh` + `StandardMaterial` из коробки — этого хватит на MVP и хороший кусок пути после.

| Сущность | Визуал | Bevy-примитив | Цветовая семантика |
|---|---|---|---|
| **Нейрон** | Сфера | `Sphere { radius }` | Тип нейрона (4-бит маска → цвет из палитры) |
| **Аксон (растущий)** | Цилиндр + конус FOV | `Cylinder` + `Cone` | Направление роста, конус = зона поиска дендритов |
| **Аксон (зрелый)** | Только цилиндр | `Cylinder` | Конус убирается после завершения роста |
| **Синапс (дендрит)** | Маленькая сфера на конце аксона | `Sphere { radius: small }` | Активен / неактивен |
| **Шард** | Прозрачный кубоид | `Cuboid` + `StandardMaterial { alpha_mode: Blend }` | Загрузка шарда (зелёный → красный) |
| **Спайк-событие** | Вспышка на нейроне | Анимация `scale` + `emissive` | Яркость = сила спайка |
| **Ghost Axon** | Пунктирная линия | `Gizmo` или отдельный Mesh | Выделен штриховкой / своим цветом |

> **Почему не сторонние ассеты (glTF/fbx)?** Нейронов может быть миллионы — статичные mesh-файлы не масштабируются. Процедурные примитивы Bevy генерируются на CPU и батчатся GPU instancing'ом (`GpuInstancing`). Смена визуала на красивый glTF-ассет в будущем — это просто замена `Mesh`-источника, не переписывание архитектуры.

---

#### GPU Instancing — Масштабируемость

Миллион нейронов = миллион draw calls без instancing'а → смерть. Решение:

- Все нейроны одного типа → один `InstancedMesh` (один draw call на тип)
- Аксоны → отдельный `InstancedMesh` для цилиндров; конусы → отдельный `InstancedMesh`, рендерится только для аксонов со статусом `Growing`
- Позиции/цвета передаются через instance buffer (обновляется из snapshot'а)
- LOD: при zoom out нейроны деградируют до `Point` (billboard sprite), при zoom in — полная сфера

---

#### Управление камерой

**Камера — свободная FPS-сущность**, независимо летающая в пространстве. Нет pivot-точки, нет orbit'а.

**Режим камеры** активируется нажатием `Alt` (toggle). В активном режиме курсор захватывается. Повторный `Alt` или `Esc` — выход из режима, курсор освобождается.

| Действие | Управление | Условие |
|---|---|---|
| Активировать / деактивировать режим камеры | `Alt` (toggle) или `Esc` для выхода | — |
| Повернуть взгляд (yaw / pitch) | Движение мыши | Режим камеры активен |
| Двигаться вперёд / назад / влево / вправо | `W` `A` `S` `D` — относительно направления взгляда | Режим камеры активен |
| Подняться по глобальному Z | `Shift` | Режим камеры активен |
| Опуститься по глобальному Z | `Пробел` | Режим камеры активен |
| Pan (перемещение ⊥ взгляду по XY) | `ПКМ` зажат + движение мыши | Режим камеры активен |
| Изменить скорость движения | Колёсико мыши (множитель; не zoom) | Всегда |

> Shift, Пробел и WASD двигают камеру с **текущей скоростью** (выставленной колёсиком). Pan через ПКМ — строго перпендикулярно направлению взгляда, без изменения угла.

---

#### Визуальный стиль — «Blender без пола»

Визуальный язык слизан с Blender Viewport: тёмный фон, мягкое ambient-освещение, чистые формы без лишних эффектов.

**Вместо сетки на полу — пространственная 3D-сетка (чанки):**
- Мир разбит на чанки **10 × 10 × 10 вокселей**
- Границы чанков рендерятся как полупрозрачные линии/грани (`alpha ≈ 0.08–0.12`) — едва видимы, не мешают
- Чанки — ориентир, а не физическая сущность; совпадают с единицами координат конфига
- Рядом с камерой чанки чуть ярче, вдали — fade out по расстоянию

---

#### Режимы Отображения

| Режим | Что видно | Когда полезен |
|---|---|---|
| `Solid` | Нейроны + аксоны, цвет по типу | Просмотр архитектуры |
| `Activity` | Тепловая карта активности (cold → hot) | Live-наблюдение во время симуляции |
| `Wireframe` | Только рёбра mesh'ей | Отладка геометрии шардов |
| `Points` | Всё → точки, максимальная производительность | Огромные сети (>1M нейронов) |

#### HUD-оверлей (угол экрана)

Постоянно отображается в одном углу (по умолчанию — левый нижний):

```
Pos:   X: 142.3  Y: -88.1  Z: 55.0
Chunk: [14, -9, 5]

[нейрон выбран]
Neuron ID: 0x00A3_F219
```

- **Pos** — мировые координаты камеры (float, 1 знак после запятой)
- **Chunk** — целочисленный индекс чанка (`floor(pos / 10)`)
- **Neuron ID** — появляется только после клика ЛКМ на нейрон; пропадает при клике в пустоту

> Клик ЛКМ на нейрон = ray cast из камеры. Попадание → запросить ID из последнего snapshot'а → отобразить в HUD.

---

#### Что НЕ входит в MVP

- Выделение (selection) + inspector нейрона — следующий этап
- Анимация роста аксонов в реальном времени — следующий этап
- Кастомные шейдеры (glow, volumetric туман) — после базового рендера
- Morphing ассетов — не планируется

### 2.3 Связь с Runtime (Telemetry)

IDE подключается к runtime через **WebSocket** (не чистый TCP).

| Параметр | Значение |
|---|---|
| URL | `ws://127.0.0.1:8002/ws` |
| Транспорт | WebSocket (Binary frame) |
| Направление | runtime → IDE (однонаправленный push) |
| Периодичность | 1 кадр / BSP-батч (по окончании `sync_batch_ticks`) |

**Точный формат кадра** ([источник](file:///home/alex/Workflow/Genesis/genesis-runtime/src/network/telemetry.rs)):

```
[TelemetryFrameHeader]  — 16 байт, #[repr(C)], little-endian
  magic:        u32   = 0x_47_4E_53_53 (ASCII "GNSS", Genesis)
  tick:         u64   = номер BSP-батча
  spikes_count: u32   = кол-во u32 в payload

[Payload]
  spike_ids: u32[spikes_count]  — ID нейронов, выстреливших за батч
```

> **ID нейрона** = индекс в массиве VRAM (`tid в CUDA-ядре`). Совпадает с индексом entity в `InstancedMesh` на стороне IDE.

---

#### Позиции нейронов (загрузка при старте)

`.state` — это `ShardStateSoA`: SoA-массивы для GPU (вольтаж, флаги, дендриты...). **Отдельного массива XYZ-позиций в нём нет.**

Позиция каждого нейрона закодирована в `packed_pos: u32` (имплицитно, через `entity_seed`). **IDE не читает `.state` напрямую** — позиции получаются через Geometry Server (`TCP port+1 = 8001`).

| Шаг | Что происходит |
|---|---|
| Старт IDE | Подключаемся к `ws://127.0.0.1:8002/ws` |
| Запрос геометрии | Через `GeometryServer` (TCP 8001) — получаем позиции всех нейронов одним запросом |
| Строим World | Размещаем `InstancedMesh` сфер по полученным XYZ |
| Live | Каждый WS-кадр → `emissive` на нейронах из `spike_ids[]` |

### 2.4 Bevy как основа

**Рендер: кастомный Bevy-рендер** (не `bevy_egui`). Интерфейс через Bevy mesh/sprites/camera напрямую. Причина: интерфейс будет расти и `bevy_egui` станет ограничением раньше чем позже.

| Роль | Bevy-механизм |
|---|---|
| Состояние Layout | `Resource<WorkspaceLayout>` |
| Состояние симуляции | `Resource<SimulationState>` |
| Нейроны в 3D | `Component`-набор на entity + `InstancedMesh` |
| Приём кадра | асинх TCP-вредина → `Event<SpikeFrame>` |
| Индикация активности | По ID из `SpikeFrame` находим entity, выставляем `emissive` на 1 – 2 фрейма, затем возвращаем в 0 |
| Ввод (мышь/клавиши) | Bevy Input system |

## MVP-Картина (минимальный результат)

Что видит пользователь после первого запуска IDE:

1. Тёмный 3D-мир с полупрозрачной сеткой-чанками
2. Все нейроны отрисованы сферами на своих позициях (из `.state`), цвет по типу
3. Идёт TCP-поток с `127.0.0.1:8002` — каждый кадр вспыхивает нейроны **emission glow** по списку ID
4. Glow живёт 1–2 фрейма, затем фейдит (не требует данных от runtime кроме ID)
5. HUD с позицией камеры + номером чанка; при клике на нейрон — его ID

> Стик структуры загружается один раз (статика). Динамика = только спайк-кадры.
