# 04. Коннектом (Connectivity)

> Часть архитектуры [Genesis](../../README.md). Как нейроны находят друг друга, как связи рождаются, живут и умирают.

## 1. Правила Роста и Соединений (Connectivity)

### 1.1. Архитектура: Pub/Sub

| Роль | Аналогия | Поведение |
|---|---|---|
| **Аксон (Publisher)** | Радиостанция | Вещает сигнал в пространство (Active Tail + Global Buffer). Ему всё равно, слушает 0 или 1 000 000 дендритов. **Не хранит список слушателей** (это бы убило память). |
| **Сома/Дендрит (Subscriber)** | Радиоприёмник | Решает, на какой Axon ID «настроиться». Лимит каналов = 128 слотов. |

Ограничение в 128 - это лимит **входной пропускной способности** конкретного нейрона, а не выходной популярности аксона.

### 1.2. Классификация Аксонов (Axon Classes)

В `anatomy.toml` и `blueprints.toml` аксоны явно делятся на три класса, чтобы не считать лишнюю физику:

| Класс | Описание | Физика |
|---|---|---|
| **Local** (Интернейроны) | Локальные связи, облака торможения | Честная физика, коллизии |
| **Horizontal** (Слой) | Связи в плоскости XY | Честная физика в плоскости |
| **Projectors** (Белое вещество) | Дальние/межзональные связи | Только `Port Out` + `Port In` + `Delay_Counter` |

- **Оптимизация Projectors:** Симулируем только точку входа (в кабель) и точку выхода. Ствол аксона **не существует** в физике коллизий (экономим память и такты) - он существует только как `Delay_Counter`.

### 1.3. Аксоны (Axon Growth)

- **Цель (Target):** Не "вектор к центру" (избегать пучка в одной точке), а движение строго по Z (вверх/вниз) к целевому слою.
- **Глубина (Penetration):** Аксон должен не просто коснуться границы слоя, а проникнуть внутрь (на случайную или заданную глубину внутри слоя `axon_target_level`).
- **Остановка (Stop Condition):**
  - Рассчитывается относительное положение сомы в домашнем слое: `Soma_Rel_Z = (Soma_Z - Layer_Min) / Layer_Height`.
  - Аксон останавливается в целевом слое на той же относительной позиции: `Target_Z = Target_Layer_Min + (Soma_Rel_Z * Target_Layer_Height)`.
- **Ориентация:** Растет направленно, ориентируясь по другим сомам.
- **Дифференциация роста (H/V):**
  - Вертикальные нейроны (`growth_vertical_bias ≥ 0.5`): цель = целевой слой по Z. Остановка при достижении Target_Z.
  - Горизонтальные нейроны (`growth_vertical_bias < 0.5`): цель = радиальное направление в XY-плоскости домашнего слоя. Остановка при выходе из домашнего слоя по Z или исчерпании шагов.
- **Идентификация (Identification):** Каждый сегмент аксона хранит **полную 4-битную маску типа** нейрона-владельца (Type Index 0..15, см. [03_neuron_model.md §2](./03_neuron_model.md)).
  - Это позволяет дендритам применять специфические правила пластичности (разные Learning Rates) в зависимости от типа источника, без обращения к соме владельца.

### 1.4. Терминальная Арборизация (Terminal Arborization)

Рост аксона - двухфазный процесс. Это биологически точная механика ветвления, которая радикально увеличивает шансы дендритного контакта без какой-либо нагрузки на GPU в Day Phase.

#### Фаза А: Ствол (Trunk / Projection)

Аксон растёт от сомы к целевому слою (`Target_Z`) строго направленно. Доминирует вектор `V_global` - аксон летит к цели, минимально отклоняясь. Это фаза Cone Tracing, описанная в §4.

#### Фаза Б: Крона (Terminal Arborization)

При достижении `Target_Z` поведение кардинально меняется:
- **`V_global` отключается** - глобальная цель достигнута.
- **`V_noise` (джиттер) максимизируется** - аксон переступает в режим хаотичного петляния.
- Кончик аксона блуждает в пределах `arborization_radius_um`, создавая плотное облако из 50–100 сегментов внутри целевого слоя.

```
Фаза А (Ствол):                     Фаза Б (Крона):
                                    
  Сома                                ╔═══════════╗
    │ ↑ V_global                      ║ ≈≈≈≈≈≈≈≈  ║ ← Целевой слой (Target_Z)
    │                                 ║  ≈≈≈≈≈≈   ║
    │ (Cone Tracing)                  ║   ≈≈≈≈    ║  ← Плотное облако сегментов
    │                                 ║    ~~     ║    в пределах arborization_radius
    └──────────────────→ Target_Z     ╚═══════════╝
```

#### Нулевая Стоимость для GPU

Геометрия 3D-вокселей (и арборизация в том числе) существует **исключительно на CPU** - в фазу Baking и Ночного Сна. GPU в Day Phase видит аксон как **плоский 1D-сдвиговый регистр `BurstHeads8`** - ему безразлично, прямой ствол или запутанная крона.

Вся работа с арборизацией выполняется в `genesis-baker` ночью, результат запекается в `shard.paths`. Для горячего GPU-цикла стоимость = 0.

#### Эффект: Взрывной Рост Коннективности

Без арборизации аксон оставляет в `SpatialGrid` 2–5 пролётных сегментов в целевом слое. Дендриту нужно случайно «наткнуться» на эти единицы.

С арборизацией аксон оставляет плотное облако из **50–100 контактов**. Дендрит из целевого слоя почти гарантированно попадает в поиске на один из них - шанс контакта возрастает в 20–50 раз при нулевой стоимости в рантайме.

### 1.5. Дендриты (Dendrite Connectivity) и En Passant Synapses

- **Sprouting Candidate Search:** Растущий дендрит во время Ночной Фазы ищет контакты не с кончиками аксонов (tips), а с **любым из их сегментов** (En Passant synapses). 
- **Axon Segment Grid (O(K) Spatial Hashing):** Поиск осуществляется через 3D Spatial Hash Grid, в который загружена полная история путей аксонов (из `shard.paths`). Это убивает рандомный O(N) перебор, заменяя его на **O(K) Spatial Hashing** (Zero-Cost Spatial Search).
- **Hebbian Structural Rule:** Дендрит прорастает (ищет новые синапсы) **только если его сома была активна днём** (`flags[i] & 0x01 != 0`). Это отсекает 90-99% холостых итераций CPU во время Ночной Фазы.

**Sprouting Density Invariant (The Dense Rule):**
Во время поиска свободного слота для нового синапса, цикл на CPU **обязан идти вперёд (0..127)**. Ядро `sort_and_prune` на GPU уплотняет живые связи в начале массива. Первый встреченный нулевой `target` — это гарантированный конец плотного блока и единственно верное место для вставки. Запись в конец массива (например, слот 127) сделает синапс невидимым для Day Phase из-за оптимизации Early Exit (`if (target == 0) break`).

**Dead on Arrival Protection (Стартовый капитал):**
При рождении синапса его начальный вес (`initial_synapse_weight`) автоматически сдвигается в домен массы (`<< 16`). Если полученное значение оказывается меньше порога прунинга (`prune_threshold << 16`), синапсу принудительно выдается дополнительный "капитал" (вес увеличивается), чтобы он гарантированно пережил первую Ночную Фазу и успел проявить себя в обучении.

- **Связь:** В `PackedTarget` записывается `Axon_ID` (24 бита) и `Segment_Offset` (8 бит). Это позволяет GPU в Day Phase мгновенно проверять попадание дендрита в `Active Tail` конкретного участка аксона.
- **Типовая фильтрация (Hard Filter):**
  - Дендрит проверяет полную маску типа владельца аксона (через Segment ID) до создания связи.
  - Правило: Не создавать синапсы с несовместимыми типами (whitelist/blacklist). Это предотвращает создание бесполезных связей.
- **Лимиты (Limits):**
  - **Max Synapses:** Максимум 128 дендритов на одну сому.
  - **Rule of Uniqueness:** Сома может иметь **только одну связь** с конкретным Axon ID. Дублировать сигнал бессмысленно - дефицитные слоты (128) лучше потратить на разные источники.

### 1.6. Компенсация Силы (Super-Synapse)

Отсутствие мульти-коннекта компенсируется широким динамическим диапазоном веса (`i16`: от -32768 до +32767). Знак бейкается при создании из `is_inhibitory` в `blueprints.toml`: (+) = Excitatory, (-) = Inhibitory. Абсолютное значение начального веса задаётся per-type через `initial_synapse_weight`.

- **Логика:** В момент, когда дендрит видит активный аксон: `Soma_Voltage += Synapse_Weight` (GPU: `v += (int32_t)dendrite_weights[col_idx]`, cast i16→i32 для безопасного суммирования 128 слотов).
- **Масштаб:**
  - Слабая связь (только родилась): Вес = `initial_synapse_weight` (по умолчанию `74`).
  - Сильная связь (обучилась): Вес = до `32 767` (i16 max). Добавляем `+32 767` к вольтажу.
- **Эквивалентность:** Одна прокачанная связь (32767) ≈ 443 слабых (74 × 443 ≈ 32767). Один «супер-синапс» математически заменяет пучок слабых.
- **Закон Дейла:** Знак синапса **неизменен** после создания. GSOP модифицирует только абсолютное значение: `new_abs = abs(w) + delta`, результат = `sign(w) × new_abs`. Clamp: `0 ≤ abs ≤ 32767`.
- **Почему это правильно:** В биологии 5 синапсов нужны для надёжности (вероятностный выброс медиатора). У нас физика **детерминированная** - 5 связей по 10 = 1 связь на 50.

**Жесткий инвариант Закона Дейла (Night Phase Sprouting):**

Когда во время Ночной Фазы дендрит захватывает новый аксон (через O(K) Spatial Hash Grid), знак начального веса `initial_synapse_weight` определяется **исключительно типом нейрона-владельца аксона** (Sender-Side Type), а не типом принимающей сомы.

Высокопроизводительная (DOD) реализация извлекает 4-битный тип напрямую из маски геометрии или Spatial Grid (`SegmentRef.type_idx`). Если тип аксона `is_inhibitory = true`, дендрит обязан записать **отрицательный вес** (например, `-74`). Это гарантирует, что **тормозные нейроны никогда не смогут образовать возбуждающий синапс**. Нарушение этого инварианта - признак критической ошибки в Baker или Memory Corruption.

```rust
// genesis-baker/src/bake/sprouting.rs
fn create_new_synapse(
    axon_id: usize,
    axon_tips: &[u32],  // Packed UVW координаты + 4-бит тип
    cfg: &SprintingConfig,
) -> i16 {
    // Extract type (верхние 4 бита)
    let type_bits = (axon_tips[axon_id] >> 28) & 0xF;
    let is_inhibitory = (type_bits >> 1) & 1 != 0;  // Бит 1 типа = E/I флаг
    
    // Знак ВСЕГДА зависит от аксона, НИКОГДА от дендрита
    let sign = if is_inhibitory { -1 } else { 1 };
    let initial_weight = cfg.initial_synapse_weight as i16;
    
    sign * initial_weight  // Если is_inhibitory: -74, иначе: +74
}
```

### 1.7. Стратегия Выбора (Baking Heuristics)

При плотности 4–5% и длинных аксонах количество кандидатов на подключение превышает лимит (128). На этапе Baking (CPU) применяется эвристика выбора лучших кандидатов, а не рандом:

1. **Distance Bias:** Ближайшие аксоны приоритетнее (короткие связи энергетически дешевле).
2. **Type Matching:** Реализация через `dendrite_whitelist` (жёсткий фильтр) и `sprouting_weight_type` (мягкий скоринг).

#### 1.7.1. Sprouting Score (Горячие слоты, Night Phase)

Во время Night Phase (Sprouting) дендрит выбирает аксон-кандидат по тройному скорингу. Веса конфигурируются по зоне:

```rust
// genesis-baker/src/bake/sprouting.rs

/// Готовится 1 раз за Night Phase до Sprouting.
/// Использует веса из dendrite_weights[], уже скачанных с GPU (Step 2: Download).
fn compute_power_index(soma_id: usize, weights: &[i16], padded_n: usize) -> f32 {
    let mut power = 0u32;
    for slot in 0..128 {
        let w = weights[slot * padded_n + soma_id];
        power += w.unsigned_abs() as u32; // Без float, без бранчей
    }
    power as f32 / (128.0 * 32767.0) // нормализация 0.0..1.0
}

fn sprouting_score(
    dist: f32,
    target_power: f32, // compute_power_index() целевой сомы
    epoch_seed: u64,   // wyhash(dendrite_id, master_seed + night_epoch)
    cfg: &SproutingWeights,
) -> f32 {
    let dist_score  = 1.0 / (dist + 1.0);                   // Ближе → выше
    let power_score = target_power;                          // Сильные сомы магнитят
    let explore     = random_f32(epoch_seed);                // Ротация предпочтений по эпохе

    dist_score  * cfg.weight_distance
        + power_score * cfg.weight_power
        + explore     * cfg.weight_explore
}
```

| Компонент | Описание | Типичное значение |
|---|---|---|
| `weight_distance` | Ближайший аксон = лучше | 0.5 |
| `weight_power` | Мощные сомы (Главный хуб) притягивают больше | 0.4 |
| `weight_explore` | Шум / ротация по эпохе, чтобы не зацикливаться на одних | 0.1 |
| `weight_type` | Совпадение типа источника и цели (мягкий аттрактор) | 0.2 |

> **Самостабилизация:** `soma_power_index` = ×|weight| постоянно колеблется. Сома, потерявшая связи, становится менее привлекательной. Естественная насыщаемость защищена правилом: 1 дендрит = 1 сома + `max_dendrites = 128`.

## 5. Roadmap: Дистилляция Топологии (Topology Distillation)

> [!NOTE]
> **[Planned: Концепт]** 
Реализовать механизм «выжимки» обученного графа для создания оптимизированных иерархий.

### 5.1. Дистилляция Пути (Path-Based Extraction)
- **Алгоритм**: Начинаем с выходов (Motor Readout) и идем назад по графу (Back-tracing) только по сильным возбуждающим связям (`abs(w) > threshold`).
- **Захват Контекста**: Для каждого выбранного возбуждающего нейрона автоматически подтягиваются все подключенные к нему тормозные интернейроны (PV/Sst), которые обеспечивают его стабильность.
- **Результат**: Очищенный от «шума» и неиспользуемых нейронов коннектом, содержащий только функциональный скелет навыка.

### 5.2. Слияние Ансамблей (Ensemble Merging/Stitching)
- **Идея**: Возможность объединения дистиллированных топологий из разных запусков в одну цельную зону.
- **Механика**: Baker получает на вход не только `blueprints.toml`, но и список `.state` файлов-доноров. Он «сшивает» их, выравнивая входные и выходные матрицы, создавая «Супер-Агента» с набором готовых навыков.

### 5.3. Дальнейшие идеи для развития:
1. **Dopamine-Gated Distillation**: Автоматический выбор для дистилляции только тех фрагментов сети, которые были активны в моменты получения максимального вознаграждения (`global_reward`).
2. **Quantization Optimization**: Сжатие весов дистиллированной сети из `i16` в `i8` или `ui4` для экстремального ускорения инференса «замороженных» навыков.
3. **Sub-graph Grafting**: Возможность «привить» дистиллированный слой (например, HiddenCortex с навыком баланса) в совершенно другую симуляцию как готовый модуль.

#### Механика

Аксон растёт пока не достигнет границы своего шарда (`ShardBounds`). При пересечении:

1. Генерируется **GhostPacket** - описание аксона в точке пересечения
2. Шард B получает пакет и вызывает `inject_ghost_axons()`
3. **Ghost Axon** продолжает рост оставшиеся `remaining_steps` с сохранением инерции (`entry_dir`)
4. Ghost Axon добавляется к общему списку аксонов шарда B и участвует в дендритном коннекте
5. Если Ghost Axon снова пересекает границу → генерируется новый пакет для шарда C

```rust
// genesis-baker/src/bake/axon_growth.rs

pub struct GhostPacket {
    pub origin_shard_id: u32,  // Откуда пришёл
    pub soma_idx: usize,       // Оригинальная сома (usize::MAX если внешний)
    pub type_idx: usize,       // Тип нейрона (для whitelist, affinity и т.д.)
    pub entry_x: u32,          // Точка входа в новый шард
    pub entry_y: u32,
    pub entry_z: u32,
    pub entry_dir: Vec3,       // Направление в момент пересечения (инерция)
    pub remaining_steps: u32,  // Сколько шагов осталось расти
}
```

#### Поведение Ghost Axon в шарде B

- **Нет `v_global`** - Ghost Axon не имеет новой цели, летит по инерции
- **Есть `v_attract`** - притягивается к нейронам шарда B (обычный Cone Tracing)
- **`soma_idx = usize::MAX`** - нет локальной сомы → GSOP не применяется (пластичность Ghost Axon идёт в шарде-источнике)
- **Дендриты цепляются** - сомы шарда B могут подключаться к Ghost Axon как к обычному

#### Монорежим (один шард)

`ShardBounds::full_world(&sim)` задаёт границы = весь мировой объём.  
Ghost Packets **никогда не генерируются**, поведение идентично текущему.  
При появлении пакетов - логируется предупреждение.

- **Телепорты:** Только для Global Connectivity (межзональные связи через Атлас, см. [06_distributed.md §3](./06_distributed.md)). Локальный рост - всегда бесшовный.

### 1.9. Living Axons (Tip Nudging)

Граф Genesis не статичен после Baking. Во время Ночной Фазы аксоны продолжают физически расти сквозь воксельную сетку, постоянно обновляя пул доступных дендритов (Structural Plasticity).

Чтобы этот процесс не уничтожил CPU при миллионах аксонов, применяется паттерн **Activity-Based Nudging**:

#### Logic: Когда Растёт Аксон?

1. **Локальные аксоны (Activity Gate):**
   Baker Daemon читает массив `soma_flags` (доступен через `flags_offset` в SHM v4, см. [07_gpu_runtime.md §1.7](./07_gpu_runtime.md#17-shm-binary-contract-night-phase-ipc-v4)).
   
   Если 0-й бит сомы равен 1 (`flags & 0x01 == 1` - нейрон спайковал днём), **кончик её аксона (Tip) вычисляется** через `step_and_pack` (сдвиг на 1 сегмент).
   
   Если нейрон молчал - CPU **пропускает его аксон** (O(1) без работы).

2. **Ghost-аксоны (Inertial Nudging):**
   Ghost-аксоны не имеют локальной сомы (`soma_idx = usize::MAX`). Они сдвигаются **каждую ночь безусловно** по вектору своей инерции (`forward_dir`), пока их счётчик `remaining_steps` не достигнет нуля.

3. **Persistence:**
   Когда аксон делает шаг роста (`nudging`) в Ночную Фазу:
   1. Читается `len = lengths[axon_id]`.
   2. Если `len < 256`, новый сгенерированный `PackedPosition` пишется в `paths[axon_id * 256 + len]`.
   3. `lengths[axon_id] += 1`.
   4. Новый сегмент атомарно добавляется в `AxonSegmentGrid` для доступности новым дендритам в том же тике.

#### Структура данных (CPU-side, Night Phase)

```rust
// genesis-baker/src/bake/growth.rs

/// Living Axon контролируется во время Night Phase
pub struct LivingAxon {
    pub axon_id: usize,              // Индекс в массиве axon_tips
    pub soma_idx: usize,             // usize::MAX для Ghost-аксонов
    pub tip_uvw: u32,                // Packed UVW + type (верхние 4 бита)
    pub forward_dir: Vec3,           // Инерция / направление роста
    pub remaining_steps: u32,        // Сколько шагов осталось расти
    pub last_night_active: bool,     // Флаг активности на прошлой ночи
}

fn nudge_living_axons(
    living: &mut [LivingAxon],
    soma_flags: &[u8],
    spatial_grid: &mut SpatialGrid,
) {
    for axon in living.iter_mut() {
        // Decision: расти ли в эту ночь?
        let should_grow = if axon.soma_idx == usize::MAX {
            true  // Ghost-аксон: растёт всегда
        } else {
            // Локальный аксон: только если сома спайковала днём
            soma_flags[axon.soma_idx] & 0x01 != 0
        };
        
        if should_grow && axon.remaining_steps > 0 {
            // Step: сдвиг на 1 сегмент по инерции
            axon.tip_uvw = step_and_pack(
                axon.tip_uvw,
                axon.forward_dir,
                /* gravity */ None,  // На CPU нет гравитации, только инерция
            );
            axon.remaining_steps -= 1;
            
            // Grid update: добавить новый сегмент в Spatial Grid
            spatial_grid.insert(axon.axon_id, axon.tip_uvw);
        }
    }
}
```

#### Эффект: Data-Oriented Оптимизация

- **Селективность:** Только ~10–30% аксонов активны в любую ночь (зависит от стимуляции). Остальные пропускаются за O(1).
- **Минимум CPU:** Одна ночь (~50 мс симулировано) обновляет ~10K–50K активных аксонов, 0-стоп для остальных.
- **Строгая Причинность:** Аксон растёт **только если его сома была активна**. Это естественная связь между активностью и структурной пластичностью.
- **Spatial Coherence:** Обновленный Spatial Grid гарантирует, что Sprouting (дендритный коннект) немедленно видит новые расученные сегменты и может подключиться к ним.

*Этот DOD-паттерн гарантирует, что вычислительные ресурсы тратятся только на "живые", активные участки нейросети, позволяя графу пластично менять топологию вслед за паттернами стимуляции.*

## 2. Пластичность и Математика (Plasticity & Math)

### 2.1. Вес Синапса

- **Тип:** `i16` (-32768..+32767). Знак определяет E/I (бейкается при создании связи).
- **Семантика:** Вес = сила тока со знаком. При срабатывании: `Soma_Voltage += Weight`. Знак (+/-) определяет E/I, бейкается при создании связи.
- **Инициализация:** Начальный вес задаётся в конфиге (`initial_synapse_weight` per-type из blueprints.toml).

### 2.2. Триггер Обучения: Active Tail Check

- **Правило:** При спайке сомы проверяем, касается ли дендрит **активного хвоста** (Active Tail) сигнала в аксоне (см. [05_signal_physics.md](./05_signal_physics.md)).
- **Причинность:** Сигнал физически пришёл → сома выстрелила → GSOP сработал. Не «совпадение в тике», а проверка пространственного перекрытия.
- **Результаты:**

| Ситуация | Active Tail | Действие |
|---|---|---|
| Сома спайкует, дендрит дал соме вольтаж | ✅ Да | `Weight += potentiation` (усиление) |
| Сома спайкует, дендрит НЕ дал соме вольтаж | ❌ Нет | `Weight -= depression` (ослабление) |
| Вес падает ниже `1` | - | Связь удаляется (Pruning) |

### 2.3. GSOP через LUT (Variant-Dependent Learning)

Конкретные значения GSOP **не хардкодятся**. Они задаются в [02_configuration.md §6](./02_configuration.md) и загружаются в `__constant__` память GPU как LUT по Type ID (строго 16 типов из `blueprints.toml`):

| Type Index | Potentiation | Depression | Множитель | Профиль |
|---|---|---|---|---|
| `0..15` | из конфига | из конфига | `1.0` | Стандартная пластичность (настраивается per-type) |

```
// GPU (псевдокод)
let gsop = GSOP_LUT[type_id];
weight += gsop.potentiation * gsop.multiplier;
```

### 2.4. Кривые Инерции (Inertia Curves)

Вместо бинарной заморозки (Permanence bit) - **нелинейное сопротивление изменениям**.

- **Механика:** Текущий `abs(вес)` (`i16`) делится на **16 рангов** (по 2048 единиц). Кривая задаётся через параметр `inertia_curve: [u8; 16]` в конфиге (per-type).
- **Суть:** Чем сильнее связь, тем сложнее её изменить (как усилить, так и ослабить). «Детские» связи пластичны, «старые» - монументальны.

| Ранг | Диапазон abs(вес) | Коэффициент | Поведение |
|---|---|---|---|
| 0–3 | 0 – 8 191 | Высокий | Быстрое обучение, быстрая смерть |
| 4–7 | 8 192 – 16 383 | Средний | Стабилизация |
| 8–11 | 16 384 – 24 575 | Низкий | Трудно изменить |
| 12–15 | 24 576 – 32 767 | Минимальный | Почти монументальная связь |

```
// GPU (псевдокод)
let rank = abs(weight) >> 11;                // 16 рангов (по 2048 единиц каждый)
let inertia = p.inertia_curve[rank];         // Коэффициент из constant memory (per-type)
let delta = gsop.potentiation * inertia;   // Чем выше ранг, тем меньше delta
weight = sign(weight) * clamp(abs(weight) + delta, 0, 32767);
```

**Почему не Permanence bit:** Бинарная заморозка создавала стеклянный потолок - связь, став «постоянной», теряла способность адаптироваться. Inertia Curves обеспечивают плавный переход от пластичности к стабильности без абсолютных границ.

### 2.5. Асимметричная Нейромодуляция (D1/D2 Affinities)

Механика градиентного распада по слотам (LTM/WM) вырезана для экономии тактов ALU и освобождения места в 64-байтном профиле нейрона. 
Вместо нее внедрена асимметричная реакция на глобальный дофамин. Каждый тип нейрона имеет `d1_affinity` (усиление LTP при награде) и `d2_affinity` (подавление LTD при награде). Математика интеграции описана в `05_signal_physics.md`.

### 2.6. Ночная Сортировка (Nightly Sort)

Во время фазы «Ночь» (CPU, см. [07_gpu_runtime.md](./07_gpu_runtime.md)) выполняется консолидация памяти:

1. **GPU Sort & Prune:** Segmented Radix Sort по `abs(weight)` descending. Слоты с `abs(w) < threshold` обнуляются.
2. **Неявный Promote / Evict:** Поскольку пустые слоты (target=0) гарантированно опускаются в конец массива, а живые связи сортируются по убыванию силы, сильные связи автоматически занимают слоты `0..ltm_slot_count` (LTM), а слабые вытесняются в хвост `ltm_slot_count..127` (WM). Отдельных проходов Promote и Evict не требуется.
3. **Результат:** После ночи LTM содержит `ltm_slot_count` самых сильных связей, WM - свежие кандидаты.

## 3. Мультизонная Архитектура (Multi-Zone Connectivity)

Genesis моделирует мозг как набор **зон** (Zones) - функционально различных регионов с собственной анатомией, типами нейронов и конфигурацией. Зоны связываются друг с другом через Ghost Axons. Ключевой инвариант: **для ядра (GPU-кода) не существует понятия «другая зона» или «другой сервер»**. Все различия скрыты за абстракцией Channel.

### 3.1. Иерархия: Zone → Shard → Node

| Понятие | Описание | Масштаб |
|---|---|---|
| **Simulation** | Глобальные правила физики: единый `simulation.toml` на всю симуляцию (tick_duration, voxel_size, sync_batch, master_seed). BSP-синхронизация требует единый тик. | 1 |
| **Zone** | Функциональная область мозга (V1, V2, A1, Hippocampus). Имеет собственные `blueprints.toml`, `anatomy.toml`. Параметры мира (`width_um`, `depth_um`, `height_um`) - глобальны (~simulation.toml), но каждая зона получает **смещение** и **резку** (квадрат в пространстве мира в koordinатах XYZ). | Десятки |
| **Shard** | Плоская плитка (Tile) внутри зоны. Возникает при разбиении зоны по XY (см. [06_distributed.md §1](./06_distributed.md)). Маленькая зона = 1 шард. Большая зона = N×M шардов. | Единицы–сотни |
| **Node** | Физическая единица исполнения: 1 GPU + RAM. Один Node может содержать множество шардов разных зон. | Единицы |

> - Шард не знает, какой канал используется - все три реализуют один `Channel` trait.

### 3.2. Три Канала Коммуникации (Channel Trait)

Шард не знает, где физически находится его сосед. Он взаимодействует через абстракцию **Channel**, которая имеет три реализации:

| Channel | Когда | Транспорт | Латентность | Копирование |
|---|---|---|---|---|
| **IntraGPU** | Зоны на одном GPU | Указатели в VRAM | ~0 | Нет. Обе зоны видят одну и ту же физическую память. |
| **IntraNode** | Шарды на разных GPU одной машины | NVLink / PCIe P2P | ~мкс | Аппаратный DMA между GPU. |
| **InterNode** | Шарды на разных машинах | UDP (Fast) / TCP (Slow) | ~мс | Сериализация → сеть → десериализация. Батчинг компенсирует задержку. |

**Все три реализации предоставляют одинаковый интерфейс:**

```rust
trait Channel {
    // Fast Path: синхронизировать спайки между зонами на каждом sync_batch
    fn sync_spikes(&mut self, zones: &mut [ZoneRuntime]) -> anyhow::Result<()>;
    
    // Slow Path: синхронизировать геометрию аксонов (Night Phase, Sprouting)
    fn sync_geometry(&mut self) -> anyhow::Result<()>;
    
    // Runtime Query: проверить статус канала (для debug и метрик)
    fn channel_type(&self) -> ChannelType;  // IntraGPU | IntraNode | InterNode
}

enum ChannelType {
    IntraGPU,    // Shared VRAM: прямое копирование указателей
    IntraNode,   // NVLink/PCIe: аппаратный DMA между GPU
    InterNode,   // UDP/TCP: сериализация и сеть
}
```

**Пояснение сигнатур:**
- `sync_spikes(&mut self, zones: &mut [ZoneRuntime])`: Метод принимает ссылку на все зоны runtime для доступа к их VRAM-буферам. Реализация зависит от канала:
  - IntraGPU: копирует `axon_heads[]` напрямую в VRAM
  - IntraNode: использует CUDA Peer-to-Peer memcpy
  - InterNode: сериализует, отправляет UDP, десериализует
- `sync_geometry(&mut self)`: Синхронизирует изменения слоя аксонов (Sprouting, Pruning) во время Night Phase

### 3.3. IntraGPU Channel: Единый VRAM, Изолированные Буферы

Самый частый сценарий - несколько зон на одном GPU. Каждая зона владеет своим набором SoA-буферов (`soma_voltage`, `dendrite_weights`, `axon_heads`, ...) и своим `ConstantMemory` (blueprints).

**Ключевой инвариант:** GPU-ядро **никогда** не обращается к чужой памяти. Каждая зона запускает одно и то же ядро (`UpdateNeurons`, `PropagateAxons`, `ApplyGSOP`) со своими указателями, своим `padded_n` и своим `ConstantMemory`.

**Ghost Axon при IntraGPU:**

1. Ghost Axon зоны V2, принимающий сигнал из V1 - это обычный слот в массиве `axon_heads[]` **зоны V2** (не V1).
2. **CPU Sync (между sync_batch):** Копирует значения `head` из буферов V1 в ghost-слоты буфера V2.
   - **Текущая реализация (Issue #4):** Наивная O(N) схема - отдельный `cudaMemcpy` для каждого Ghost Axon. При 5120 связей между двумя зонами это 5120 отдельных DMA транзакций за каждый sync_batch (полосе пропускания = мусор).
   - **Planned (Issue #4 решено):** Едины CUDA ядро `ghost_sync_kernel()` обрабатывает все Ghost Links за раз - O(1) по количеству связей, O(N) ядер в параллель.
3. Для GPU-ядра V2 этот ghost выглядит как обычный локальный аксон. Active Tail, GSOP, Decay - всё работает идентично.

**Текущее состояние:**
- **IntraGPU:** Функционирует корректно, но пропускная способность ограничена наивным memcpy
- **IntraNode (NVLink):** Использует аппаратный DMA, не имеет наво-памяти проблемы
- **InterNode (UDP):** Батчинг скрывает задержу сети, ~1–10 мс на sync

**Следствие:**
- Разница между IntraGPU, IntraNode и InterNode - **только в том, как CPU заполняет ghost-слоты**. GPU-ядро об этом не знает.
- На одном GPU 5 зон с разными blueprints работают так же быстро, как один монолитный шард эквивалентного размера (после оптимизации IntraGPU).
- Каждая зона может иметь свои типы нейронов, свою анатомию, свои параметры пластичности - разнообразие внутри одного GPU без оверхеда.

### 3.4. Два Потока Данных (Fast & Slow Paths)

Структура пакетов единая для всех трёх каналов. Отличается только транспорт.

**1. Активность (Fast Path / Spikes)** - каждый `sync_batch`

| Поле | Тип | Описание |
|---|---|---|
| `ghost_axon_id` | `u32` | Локальный ID Ghost Axon в буфере принимающей зоны/шарда |
| `tick_offset` | `u8` | Смещение тика внутри батча |

**2. Геометрия (Slow Path / Growth)** - Night Phase

| Поле | Тип | Описание |
|---|---|---|
| `entry_point` | `(u16, u16)` | Координаты на грани (2D) |
| `vector` | `(i8, i8, i8)` | Нормализованный вектор инерции |
| `type_mask` | `u8` | Type ID для фильтрации дендритами |
| `remaining_length` | `u16` | Оставшиеся шаги роста |

Оптимизация: при Handshake (создание Ghost Axon) стороны договариваются о `ghost_axon_id`. В горячем цикле - только лёгкий `u32`, без глобальных UUID.

### 3.5. Ghost Axons (Жизненный Цикл)

**Инвариант:** Физика внутри шарда не знает, что аксон начался в другой зоне.

- **Сущность:** Ghost Axon - полноценный аксон без локальной Soma. Его «тело» - порт на границе зоны.
- **Жизненный цикл:**
  1. **Рождение:** Аксон зоны A пересекает границу → `GhostPacket` / `AxonHandover` → зона B создаёт Ghost Axon.
  2. **Активность:** Спайк зоны A → `SpikeBatch` → `axon_heads[ghost_id] = 0` в зоне B.
  3. **Пропагация:** `PropagateAxons` двигает Ghost Axon как обычный. `UpdateNeurons` проверяет Active Tail - математика идентична.
  4. **Смерть:** Зона A прунит аксон → `PruneAxon(ghost_id)` → зона B освобождает слот.
  5. **Цепочка:** Ghost Axon может выйти за границу зоны B → экспорт в зону C.


### 3.6. Конфигурация Связей (Коннектом)

В `brain.toml` связи между зонами указываются через секцию `[[connection]]` (см. [02_configuration.md §7.4](./02_configuration.md)):

```toml
[[connection]]
from = "V1"
to = "V2"
output_matrix = "v1_output"       # Имя выходной матрицы V1
width = 64                         # Разрешение проекции
height = 64
entry_z = "top"                   # На каком слое появляются входы в V2
target_type = "All"               # К каким типам нейронов
growth_steps = 1000               # Максимум итераций Cone Tracing для Ghost Axon

[[connection]]
from = "V2"
to = "Motor"
output_matrix = "v2_output"
width = 32
height = 32
entry_z = "mid"
target_type = "All"
growth_steps = 800
```

**Механизм реализации:**

1. **Baking:** При сборке (genesis-baker) для каждой `[[connection]]`:
   - Экспортирует аксоны из `from` зоны, которые пересекают границу → `exit_points` файл
   - Импортирует Ghost Axons в `to` зону → инициирует Cone Tracing для синтеза дендритных связей
   - Результат: бинарный файл `baked/connections/{from}_{to}.ghosts` с маппингом `self_soma_id → target_ghost_axon_id`

2. **Runtime:** При инициализации (genesis-runtime):
   - Загружает все `.ghosts` файлы из папки `baked/connections/`
   - Создаёт `IntraGpuChannel` для каждой пары (dari [03 т04 §3.3](./04_connectivity.md#33-intragpu-channel-единый-vram-изолированные-буферы))
   - Ghost Axons физически размещаются в `axon_heads[]` принимающей зоны (как обычные аксоны)

> [!IMPORTANT]
> В будущих версиях вместо MVP-заглушки число связей будет **динамически вычисляться** при Baking на основе плотности нейронов и топологии, обеспечивая биологически реалистичные проекции (всё-на-всё, слой-на-слой, мозаичные паттерны).

## 4. Алгоритм Роста: Cone Tracing & Steering

Аксон не растёт слепым лучом (Raycast). Это автономный агент, использующий эвристику **Sensing → Weighting → Steering** для максимизации потенциальных связей. Тяжёлая математика, но выполняется **один раз** на этапе Baking (CPU). В GPU-память попадает готовый массив координат.

### 4.1. Фаза 1: Сенсорика (Sensing)

На каждом шаге роста кончик аксона (Head) сканирует пространство перед собой.

- **Инструмент:** Конический запрос (Cone Query) через Spatial Hash на базе PackedPosition (X|Y|Z).
- **Параметры:**
  - `FOV` (Field of View): Угол конуса (например, 60°). Хранится как `fov_cos = cos(FOV/2)` - без тригонометрии в цикле.
  - `Lookahead` (`MAX_SEARCH_RADIUS`): Дистанция «зрения» (например, 50 мкм).
- **Фильтрация:** В выборку попадают только сомы совместимых типов (проверка маски `Type_ID` из `blueprints.toml`). Игнорируются сомы, уже имеющие связь с этим аксоном.
- **Сложность:** O(K) - ищем только в соседних чанках Spatial Grid, не перебираем все N нейронов.

### 4.2. Фаза 2: Взвешивание (Weighting)

Вычисляем вектор локального притяжения (`V_attract`). CPU-only, f32 легален.

```rust
// baking_compiler/src/cone_tracing.rs

pub fn calculate_v_attract(
    head_pos: Vec3,       // f32, текущая координата конца аксона
    forward_dir: Vec3,    // f32, нормализованный вектор роста
    fov_cos: f32,         // cos(FOV / 2), вычислен при старте
    spatial_grid: &Grid,  // Ключи из X|Y|Z PackedPosition
) -> Vec3 {
    let mut v_attract = Vec3::ZERO;
    let mut total_weight = 0.0;

    // 1. Поиск кандидатов в соседних чанках: O(K), не O(N)
    // Защита от OOB: Grid обязан делать clamp(0, MAX) при поиске чанков
    let candidates = spatial_grid.get_in_radius_safe(head_pos, MAX_SEARCH_RADIUS);

    for target in candidates {
        let dir = target.pos - head_pos;
        let dist_sq = dir.length_squared();
        if dist_sq > MAX_RADIUS_SQ { continue; }  // Без sqrt на этапе отсева

        let dist = dist_sq.sqrt();
        let dir_norm = dir / dist;

        // 2. Проверка конуса: Dot Product (без atan2/sin/cos)
        if forward_dir.dot(dir_norm) >= fov_cos {
            // Обратный квадрат расстояния = закон притяжения
            let weight = target.attraction_gradient / (dist_sq + 1e-5);
            v_attract += dir_norm * weight;
            total_weight += weight;
        }
    }

    // 3. Итоговый вектор коррекции
    if total_weight > 0.0 {
        (v_attract / total_weight).normalize()
    } else {
        forward_dir  // Нет целей → расти прямо
    }
}
```

### 4.3. Фаза 3: Руление и Квантование (Steering & Packing)

Итоговый вектор роста - смешение глобальной цели и локального интереса:

```
V_next = Normalize(V_global * W_inertia + V_attract * W_sensor + V_noise * W_jitter)
```

| Компонент | Описание | Роль |
|---|---|---|
| `V_global` (Goal) | Вектор к целевому слою (по Z / топологии проекции) | Гарантирует, что аксон дойдёт до цели |
| `V_attract` (Local) | Тяга к скоплениям нейронов (§4.2) | Создаёт извилистость (tortuosity) |
| `V_noise` (Jitter) | Шум (`WyHash64(master_seed + axon_id + step)`) | Выход из локальных минимумов |

- **Веса (`W_*`):** Задаются в `anatomy.toml`. Инерция 60–80%.

**Дискретный Шаг и Упаковка (CPU Night Phase):**

Математика ведётся в f32, но в память запекается квантованный `u32` (PackedPosition). Позиция f32 сохраняется между шагами для защиты от drift округления.

```rust
// baking_compiler/src/axon_growth.rs

pub fn step_and_pack(
    current_f32_pos: &mut Vec3,   // f32 позиция - предотвращает drift
    v_global: Vec3,
    v_attract: Vec3,
    v_noise: Vec3,
    weights: &SteeringWeights,
    owner_type_mask: u8,          // 4 бита (Geo|Sign|Variant)
    segment_length_voxels: f32,   // 2.0 (50 мкм / 25 мкм)
) -> u32 {
    // 1. Steering (f32)
    let v_steer = (v_global * weights.global
                 + v_attract * weights.attract
                 + v_noise * weights.noise).normalize_or_zero();

    // 2. Step
    *current_f32_pos += v_steer * segment_length_voxels;

    // 3. Quantize → voxel coords (с защитой границ)
    let x = (current_f32_pos.x.round() as u32).min(1023); // 10 бит
    let y = (current_f32_pos.y.round() as u32).min(1023); // 10 бит
    let z = (current_f32_pos.z.round() as u32).min(255);  // 8 бит
    let t = (owner_type_mask & 0x0F) as u32;               // 4 бита

    let packed = (t << 28) | (z << 20) | (y << 10) | x;

    // 4. Защита от Stagnation (Залипание на границе)
    // Если шаг не изменил квантованную координату (уперлись в край сетки),
    // нужно прервать рост, чтобы не плодить фантомные сегменты в одном вокселе.
    assert!(packed != previous_packed_pos, "Stagnation detected: axon stalled at boundary");
    
    packed
}
```

- **4 байта на сегмент.** 1M сегментов = 4 МБ в Read-Only/Texture Memory.
- **Type под рукой:** `seg_val >> 28` → дендрит мгновенно фильтрует тип без обращения к соме.

### 4.4. Результат: Smart Wiring

В отличие от рандома, такой аксон:

1. **Огибает пустоты:** Если впереди нет нейронов, идёт прямо по `V_global` (экономит длину).
2. **Ныряет в кластеры:** Если видит группу нейронов, отклоняется, чтобы пройти сквозь неё, максимизируя Overlap Probability для дендритов.
3. **Формирует пучки:** Аксоны, идущие в одну зону, естественным образом сближаются, формируя тракты (как в белом веществе) - их притягивают одни и те же промежуточные цели.
