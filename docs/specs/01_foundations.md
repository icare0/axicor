# 01. Фундамент (Foundations)

> Часть архитектуры [Genesis](../../design_specs.md). Определяет фундаментальные инварианты системы. Нейронная модель: **GNM** (Genesis Neuron Model), мембранная механика: **GLIF** (Genesis Leaky Integer-Fire), пластичность: **GSOP** (Genesis Spike-Overlap Plasticity).

## 1. Пространство и Время (Units & Coordinates)

Система оперирует в гибридной системе координат для обеспечения масштабируемости архитектуры и точности физических взаимодействий.

### 1.1. Пространственная Метрика (Spatial Domain)

// - **Micro-Space (VRAM, 32-bit):** Внутри шарда координаты остаются `PackedPosition` (11-bit X, 11-bit Y, 6-bit Z). Это локальный офсет.
// - **Macro-Space (Cluster, 64-bit):** На уровне оркестратора `genesis-node` вводятся глобальные абсолютные координаты: `Global_X = Shard_Macro_X + Local_X`. 
// - **Абстракция:** GPU ничего не знает о макро-позиции. Сетевой слой при передаче GhostPacket "перешивает" локальные координаты отправителя в локальные координаты получателя, опираясь на их 64-битные макро-смещения.

- **Абсолютная единица:** `1.0` (float) = **1 мкм** (микрометр).
  - Используется для: расчета физики диффузии, длины аксонов, радиусов поиска дендритов и скоростей.
  - Почему: Это позволяет использовать реальные нейробиологические константы без магических коэффициентов.
- **Относительная единица (Нормализация):** `0.0 ... 1.0` (pct).
  - Используется для: определения границ слоев (Layers) и топологии зон.
  - Логика: Слой L1 может занимать `0.0 - 0.1` (10% высоты). Это значение умножается на `world.height` (в мкм) при инициализации. Это позволяет менять размер "мозга" без переписывания архитектуры слоев.
- **Дискретизация (Grid):** Пространство квантуется на **воксели** (размер задается в конфиге, например, 10 мкм).
  - Используется для: оптимизации коллизий и поиска соседей (Spatial Hashing).

### 1.2. Дискретизация Аксона (Axon Segmentation)

Чтобы сэкономить память, мы не храним каждый воксель пути. Аксон — это цепочка **Сегментов**.

- **Параметр:** `segment_length_voxels` (например, `4` вокселя = 100 мкм при вокселе 25 мкм).
- **Логика:**
  - Аксон растет «шагами». Один шаг роста добавляет один Сегмент в массив геометрии.
  - **Идентификация:** Сегмент — единица взаимодействия. Дендрит подключается не к координатам `(X, Y, Z)`, а к паре `Axon_ID + Segment_Index`.
  - **Хранение:** В памяти хранится только список узловых точек (начала сегментов). Пространство между ними считается прямой линией (или интерполируется) для коллизий.
- **Оптимизация:** Снижает потребление памяти в `segment_length_voxels` раз. Вместо 1000 записей на 1 мм пути (при разрешении 1 мкм) — всего **10 записей** (при сегменте 100 мкм).

### 1.3. Физика Сигнала: «Активный Хвост» (Active Tail)

Сигнал не точечный (не один бит). Это «пакет» энергии, который имеет длину.

- **Параметр:** `signal_propagation_length` (длина хвоста в сегментах).
- **Механика:**
  - Сигнал — это «поезд», который едет по рельсам (сегментам).
  - Если `signal_propagation_length = 3`, то в момент T активны сегменты `[i, i+1, i+2]`.
  - В момент `T+1` «поезд» сдвигается на скорость сигнала `v`: активны `[i+v, i+v+1, i+v+2]`.
- **Обоснование:** Гарантирует, что даже при высокой скорости сигнал не «перепрыгнет» дендрит. Дендрит успеет засечь соприкосновение с «хвостом».

**Визуализация: Движение на временной оси**

Пусть `v_seg = 1` (1 сегмент/тик), `propagation_length = 3` (3 сегмента):

```
Тик 0:  head=3  Active: [1, 2, 3]       ←──────┐
                                               |
Тик 1:  head=4  Active: [2, 3, 4]         ←────┼──┐
                                               |  |
Тик 2:  head=5  Active: [3, 4, 5]           ←──┼──┼──┐
                                               |  |  |
Сегменты: ──┴──┴──┴──┴──┴──┴──┴──→
            0  1  2  3  4  5  6  7

```

Каждый тик:
- `head` смещается на `v_seg` позиций вперёд
- Диапазон активности `[head - propagation_length + 1, head]` сдвигается вслед за главой
- Дендриты, подключённые к сегментам в диапазоне, получают сигнал

### 1.4. Временная Метрика (Temporal Domain)

- **Квант времени:** 1 Тик.
- **Разрешение:** `tick_duration_us = 100` (0.1 мс).
  - **Обоснование:** 1 мс слишком "грубая" для GSOP (теряется порядок спайков). 100 мкс — необходимый минимум для каузальности.
  - Все таймеры (рефрактерность, затухание) задаются в `u64` тиках.
  - Пример: Рефрактерность 5 мс = 50 тиков.

### 1.5. Производные Величины (Pre-calculated)

Выносим математику из горячего цикла (Hot Loop). В конфигах пишем понятные человеку величины (м/с, мкм, мс), но движок при старте пересчитывает их в «сырые» константы для GPU.

**Таблица производных величин:**

| Выходная величина | Входные параметры | Формула | Пример | Назначение |
|---|---|---|---|---|
| `signal_speed_um_tick` | `signal_speed` (м/с), `tick_duration_us` | `(speed_m_s × 1e6 мкм/м) × (tick_duration_us × 1e-6 с)` | `0.5 м/с × 100 мкс = 50 мкм/тик` | Абсолютное смещение головы сигнала за тик |
| `v_seg` | `signal_speed_um_tick`, `segment_length_um` | `signal_speed_um_tick ÷ segment_length_um` | `50 мкм/тик ÷ 50 мкм = 1 сегмент/тик` | Индекс смещения в GPU-цикле (`head += v_seg`) |
| `segment_length_um` | `voxel_size_um`, `segment_length_voxels` | `voxel_size_um × segment_length_voxels` | `25 мкм × 2 = 50 мкм` | Промежуточный расчёт для `v_seg` |
| `matrix_spatial_scale` | `zone_width_um`, `matrix_width_px`, `zone_depth_um`, `matrix_height_px` | `(width_um / width_px, depth_um / height_px)` | `(1000 мкм / 64 px, 1000 мкм / 64 px) = (15.625, 15.625)` | Масштаб проекции пикселя на территорию зоны |

**Реализация:** Все расчёты инкапсулированы в [`genesis_core::physics::compute_derived_physics()`](https://github.com/your-repo/genesis-core/src/physics.rs) — каноническая реализация вычисления производных из raw параметров.

**Конвейер пересчёта (Startup → GPU):**

Конкретный пример с конфигом:
- В `simulation.toml`: `signal_speed = 0.5 м/с`, `tick_duration = 100 мкс`
- Движок вычисляет: `signal_speed_um_tick = 50 мкм/тик`
- В GPU-память грузится число **`50`**
- Нейрон в горячем цикле прибавляет: `axon_head += 50` (не `Speed × Time`)

Для дискретной скорости (`v_seg`):
- В конфиге: `segment_length_voxels = 2`, `voxel_size_um = 25`
- Движок вычисляет: `segment_length_um = 50 мкм`
- Затем: `v_seg = 50 мкм/тик ÷ 50 мкм = 1`
- В GPU-память грузится число **`1`**
- Горячий цикл: `segment_head += 1` (чистый `u32` сдвиг)
- Active Tail (§1.3) — это диапазон `[head - propagation_length .. head]`, сдвигаемый на `v_seg` каждый тик

### 1.6. Критическая Связка Параметров (Math Check)

Скорость сигнала и длина сегмента — жёстко связанные величины. Если связь нарушена, физика ломается.

- **Инвариант:** `signal_speed_um_tick` **ОБЯЗАНА** быть кратна `segment_length_um`.
  - Скорость в сегментах/тик: `v_seg = signal_speed_um_tick / segment_length_um` → **целое число**.
  - Если `v_seg` дробное (например, 1.5 сегмента/тик) — позиция хвоста требует float-арифметики, интерполяции и усложнения шейдера. Для GPU это неприемлемо.
- **Механика:** За один тик «поезд» (Active Tail, §1.3) сдвигается ровно на `v_seg` сегментов. Никаких дробных позиций — чистый `u32` сдвиг индексов.
- **Зафиксированная конфигурация:**

| Параметр | Значение | Единица |
|---|---|---|
| `tick_duration_us` | `100` | мкс (0.1 мс) |
| `voxel_size_um` | `25` | мкм |
| `segment_length_voxels` | `2` | вокселя (= 50 мкм) |
| `signal_speed_um_tick` | `50` | мкм/тик (= 0.5 м/с) |
| **`v_seg`** | **`1`** | **сегмент/тик** |

- **Верификация:** `50 мкм/тик ÷ 50 мкм/сегмент = 1` ✓ (целое).
- **Биологическое соответствие:** 0.5 м/с — реалистичная скорость для безмиелиновых аксонов в коре.

#### 1.6.1. Порядок Вычисления v_seg

Полная цепочка вычисления дискретной скорости из параметров конфига:

**Шаг 1. Размер сегмента в микрометрах:**
```
segment_length_um = voxel_size_um × segment_length_voxels
segment_length_um = 25 мкм × 2 = 50 мкм
```

**Шаг 2. Дискретная скорость (сегменты/тик):**
```
v_seg = signal_speed_um_tick ÷ segment_length_um
v_seg = 50 мкм/тик ÷ 50 мкм/сегмент = 1 сегмент/тик
```

**Шаг 3. Верификация целочисленности:**
```
assert (signal_speed_um_tick % segment_length_um == 0)
assert (50 % 50 == 0) ✓
```

Если на шаге 3 вычисляется **не целое число** — конфигурация невалидна и должна быть отклонена при инициализации.

> **Правило:** При изменении любого из этих четырёх параметров необходимо пересчитать `v_seg` и убедиться, что он остаётся целым.

## 2. Глобальный Детерминизм (Master Seed)

Без этого отладка распределённой системы невозможна. «Плавающий баг» на кластере из 10 машин найти нереально. Если вылез баг — сохраняешь `master_seed`, и баг воспроизводится у любого разработчика на любой машине.

### 2.1. Конфигурация

- **Параметр:** `master_seed` (`String`) в `simulation.toml`.
- **Тип String:** Строка хэшируется в `u64` при старте. Это не влияет на производительность (однократная операция), но позволяет использовать читаемые сиды: `"GENESIS"`, `"HELLO_WORLD"`, `"DEBUG_RUN_42"`.
- **Правило:** Это **единственная точка входа энтропии**. Никаких `time(NULL)`, `std::random_device` или `SystemTime::now()` в коде быть не должно.

```toml
[simulation]
master_seed = "GENESIS"
```

### 2.2. Имплементация (Stateless Hashing)

Нельзя создать один генератор `Random(seed)` и дёргать `.next()`. В многопоточной среде (и тем более на разных машинах) порядок вызовов `rand()` непредсказуем — результат зависит от планировщика потоков.

- **Метод:** Хэширование, а не последовательная генерация.
- **Формула:** `Local_Seed = Hash(Master_Seed_u64 + Unique_Entity_ID)`
  - Пример: Свойства нейрона №5001 всегда вычисляются как `WyHash64(Seed + 5001)`.
  - Алгоритм: `WyHash64` — быстрый, неколлизионный, детерминированный.
- **Почему это работает:** Генерация привязана к `Entity_ID`, а не к порядку выполнения. Мир можно генерировать кусками (шардами) в любом порядке, и нейрон №5001 всегда будет одинаковым — неважно, создали его первым или миллионным.

### 2.3. Гарантии (Structural Determinism)

- **Инициализация:** Один и тот же `master_seed` + одни и те же конфиги = **бит-в-бит идентичный стартовый граф** (координаты, типы, аксоны, UV-проекции). Независимость от железа: генерация на 1 ядре CPU и на кластере из 100 GPU даёт один результат.
- **Hot Loop (День):** Внутри одного «Дня» физика GLIF и пластичность GSOP детерминированы — Integer Physics гарантирует идентичное поведение на любом GPU.
- **Эволюция:** Если запустить систему на разном железе, через сутки реального времени графы **разойдутся**. Быстрый GPU = быстрый метаболизм = больше циклов Ночи = другой опыт. **Это легализовано:** разное железо = разная жизнь существа, не баг.
- **Отладка:** Баг воспроизводится детерминированно в пределах одной конфигурации — достаточно передать `master_seed`, версию конфигов и лог `night_cycle_count`.
